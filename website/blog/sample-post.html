How We Built a Serverless Form Backend

A deep dive into FormBridge's architecture, design decisions, and the lessons we learned building a production-grade serverless form handling system on AWS.

Introduction

Contact forms are one of the most common features on websites, yet they remain surprisingly complex to build and maintain. You need a backend to handle submissions, a database to store them, email integration to notify users, and monitoring to track everything. Most solutions force you to choose between vendor lock-in (like Typeform) or managing your own infrastructure.

FormBridge was born from a simple idea: what if you could get a production-grade form backend with zero infrastructure management, free-tier eligibility, and complete customization?

Design Requirements

Before we wrote any code, we defined our key requirements:

  • Serverless: No servers to manage, auto-scale to zero
  • Free-tier eligible: Small deployments should cost $0
  • API-first: RESTful interface with clear request/response contracts
  • Rate-limited: Prevent abuse with per-key throttling
  • Secure: API keys, optional HMAC signing, encrypted storage
  • Observable: CloudWatch logs, alarms, analytics dashboard
  • Exportable: CSV exports for data portability

Architecture Overview

FormBridge is built on a classic serverless stack: API Gateway, Lambda, DynamoDB, SES, and CloudWatch.

This architecture has several advantages: individual services scale independently, you pay only for what you use, and the managed services handle reliability and security for you.

API Gateway & Routing

API Gateway serves two purposes: it's the public entry point for all requests, and it's where we enforce security policies. 

POST /submit HTTP/1.1
Content-Type: application/json
X-Api-Key: key_abcdef123456

{
  "name": "John Doe",
  "email": "john@example.com",
  "message": "Your message here..."
}

Every request must include an X-Api-Key header. API Gateway uses a Usage Plan to enforce rate limiting (100 requests/hour by default, configurable per key).

Lambda Handler

The Lambda function is where the magic happens. It receives the API Gateway event, validates the input, and orchestrates the submission workflow. 

def lambda_handler(event, context):
    # Parse and validate input
    body = json.loads(event['body'])
    name = body.get('name', '').strip()
    email = body.get('email', '').strip()
    message = body.get('message', '').strip()
    
    # Store in DynamoDB
    submission_id = store_submission(name, email, message)
    
    # Send email
    send_email(name, email, message)
    
    # Return success
    return {
        'statusCode': 200,
        'body': json.dumps({
            'id': submission_id,
            'status': 'sent'
        })
    }

Error handling is critical. Invalid input returns 400, authorization failures return 403, and server errors return 500 with CloudWatch logging for debugging.

Storage & Email

We use DynamoDB for storage because it's serverless, scales automatically, and fits the free tier beautifully.

Table schema: 

SubmissionsTable:
  PK: form_id (String)  # "website-contact"
  SK: timestamp#id (String)  # "20251105T103045Z#sub_abc123"
  Attributes:
    - name (String)
    - email (String)
    - message (String)
    - status (String): "sent" | "failed"
    - ttl (Number): Auto-delete after 90 days

For email, we use AWS SES (Simple Email Service). In development mode, submissions go to a sandbox email. In production, we use a verified domain for sending.

Security Considerations

Security is not an afterthought at FormBridge. Here are the layers we implemented:

  1. API Keys: Every request requires an API key, preventing anonymous abuse
  2. Rate Limiting: Usage Plans enforce per-key quotas
  3. HMAC Signing (Optional): Additional request authentication with HMAC-SHA256
  4. CORS: Configured to allow only your domain
  5. Encryption: All data encrypted in transit (HTTPS) and at rest (DynamoDB default)
  6. IAM Roles: Lambda has minimal permissions (least privilege)

Lessons Learned

1. Cold Starts Matter

Lambda cold starts can add 1-2 seconds to first request. We minimize this by keeping the handler lean and using Lambda layers for dependencies.

2. Monitoring is Essential

CloudWatch logs and metrics are your best friends in production. Set up alarms early for error rates, latency, and throttling.

3. Test Edge Cases

What happens with very long messages? Concurrent requests? Database throttling? Test these scenarios before production.

4. Cost Monitoring

Set up CloudWatch alarms for unexpected cost spikes. DynamoDB throttling can become expensive fast if not configured correctly.

Conclusion

Building FormBridge taught us that serverless doesn't mean careless. Thoughtful architecture, comprehensive logging, and security from the start are essential. The result is a system that's both cost-effective and production-ready.
